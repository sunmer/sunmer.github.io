Bästa verktygen för innehållshantering: Så inför du ett Git-baserat CMS till GitHub Pages (med verifierbara steg)

Ett Git-baserat CMS är ett innehållsverktyg där redigeringar blir commits i ett repository, ofta via pull requests (PR). För en statisk sajt på GitHub Pages är det ett praktiskt sätt att få “CMS-känsla” utan en egen server: du behåller versionshistorik, kan granska ändringar och publicerar via samma flöde som din kod.

Den här guiden utgår från att du vill välja bland bra innehållshanteringsverktyg (”best content management tools”) men att du specifikt vill ha en Git-baserad variant som fungerar med GitHub Pages. Fokus är en verktygsneutral procedur som du kan följa oavsett om ditt CMS gör direkta commits eller skapar PR:ar. Där det spelar roll pekar jag på etablerade Git-baserade CMS som exempel (t.ex. Decap CMS).

Vad som räknas som “bästa” innehållsverktyg i en GitHub Pages-miljö

När du publicerar via GitHub Pages är “bästa CMS” sällan det som har flest block och mallar. Det som brukar avgöra om det funkar i praktiken är:

  • Git-först: Innehållet ligger som filer (Markdown/MDX/YAML/JSON) i ditt repo och ändringar går via Git.
  • Förutsägbar publicering: Du kan tydligt se vad som deployas och var du felsöker när något inte bygger.
  • Behörigheter och spårbarhet: Minsta möjliga åtkomst, och en revisionskedja i GitHub (vem ändrade vad, när, och vem godkände).
  • Preview och granskning: Helst PR-baserat flöde så du kan läsa diffar innan publicering.

GitHub Pages kan publicera antingen från en branch/folder eller via GitHub Actions (inställningen finns under “Build and deployment” → “Source”) enligt GitHub Docs (Configuring a publishing source for your GitHub Pages site). För Git-baserade CMS blir den inställningen central, eftersom den styr var “sanningens version” av publicerat innehåll kommer ifrån.

Förberedelser: repo, branch-struktur och mappning av innehåll

Börja med att bestämma tre saker innan du ens väljer CMS-UI:

  1. Vilken site-typ du kör
  • Använder du ett repo för en användar-/organisationssajt (t.ex. username.github.io) eller en projektsajt? GitHub Pages beskriver upplägget och att du får en site per konto/organisation, samt typiska repo-namn för user/org sites (pages.github.com).
  1. Vilken publiceringskälla du vill ha
  • Deploy from a branch: GitHub bygger/deployar från vald branch + folder.
  • GitHub Actions: du styr bygget med ett workflow och deployar med Pages-actions.

GitHub själva dokumenterar hur du väljer detta i repo-inställningarna (Configuring a publishing source…). Om du vill använda en modern byggkedja (t.ex. Astro/Next export/Vite) eller behöver plugins GitHub Pages inte stödjer, hamnar du ofta i Actions-spåret.

  1. Innehållsmappning (content model) som är tydlig i filsystemet Håll det enkelt. Ett bra minimival som fungerar för många statiska generators:
  • /content/ för sidor och inlägg i Markdown
  • /content/posts/ eller Jekylls standard /_posts/ om du kör Jekyll
  • /assets/ eller /static/ för bilder
  • README.md som beskriver flödet (för redaktörer och utvecklare)

Jekyll är ett specialfall eftersom GitHub Pages har en definierad “stödd”-yta och begränsar plugins; GitHub dokumenterar att sidor inte kan byggas med plugins som inte stöds (About GitHub Pages and Jekyll / Using Jekyll plugins with GitHub Pages). Det är en vanlig anledning att istället bygga i Actions och bara deploya statiska output-filer.

Checklista: steg för steg från tomt repo till redigerbart innehåll

Checklista: steg för steg från tomt repo till redigerbart innehåll

  1. Skapa (eller städa) repository och välj publiceringsstrategi

    • Åtgärd: Skapa repo och lägg en minimal startsida, t.ex. index.html i root, eller ett generator-projekt med definierad output-mapp. Bestäm om du kör “Deploy from a branch” eller “GitHub Actions” i Settings → Pages → Build and deployment → Source.
    • Verifiering: I GitHub UI ska du se vald “Source” enligt GitHub Pages-inställningen och (när den första builden gått) en publicerad URL under Pages. GitHub beskriver exakt var inställningen finns och vad alternativen innebär (Configuring a publishing source…).

  2. Lås en tydlig branch-struktur: en publiceringsbranch och en arbetsbranch

    • Åtgärd: Bestäm main som publiceringsbranch och skapa en arbetsbranch, t.ex. content. Alternativ: om du måste deploya från en separat branch (t.ex. gh-pages) så dokumentera det i README.md.
    • Verifiering: Under repo-fliken “Branches” ser du båda brancherna. Under “Settings → Pages” ser du att rätt branch används om du kör “Deploy from a branch”.

  3. Skapa en minimal, konsekvent innehållsstruktur

    • Åtgärd: Skapa mapparna /content/ och /assets/ (eller motsvarande). Lägg en första fil, t.ex. /content/about.md och en bild i /assets/.
    • Verifiering: I repo-vyn kan du klicka dig till filerna och se att sökvägarna stämmer. Om din site redan bygger: kontrollera att sidan finns på din Pages-URL (eller i preview i PR om du har sådan).

  4. Definiera ett “CMS-kontrakt” i repo: vad får CMS:et ändra

    • Åtgärd: Skriv i README.md vilka mappar CMS får skriva till (exempel: /content/**, /assets/**) och vilka filtyper som gäller (exempel: .md, .yml). Lägg till en enkel regel: ”Ändra inte /config/ utan PR-granskning”.
    • Verifiering: En extern redaktör kan läsa README.md och förstå exakt vilka filer som ska röras. För dig: det blir en referens när du sätter branch-skydd (steg 7).

  5. Välj ett Git-baserat CMS-läge: direkt-commit eller PR-flöde

    • Åtgärd: Välj ett CMS som kan arbeta mot GitHub. Exempel: Decap CMS kan antingen pusha direkt till publiceringsbranchen eller använda “Editorial Workflow” som skapar PR:ar för opublicerade ändringar (Decap CMS – Editorial Workflows).
    • Verifiering: Gör en testredigering i CMS:et och kontrollera i GitHub om det blev (A) en commit direkt på publiceringsbranchen eller (B) en ny branch + PR. I PR-fallet ska du se en normal GitHub PR med diffar.

  6. Säkerställ minsta behörighet för CMS-åtkomst (token/app)

    • Åtgärd: Använd helst en GitHub App eller en finmaskig token med minsta möjliga repo-åtkomst. Om du måste använda en finmaskig personal access token (PAT), välj endast nödvändiga repo-permissioner och begränsa token till ett repo.
    • Verifiering: I GitHub kan du kontrollera token- eller app-setup och att den är begränsad. GitHub listar hur finmaskiga tokens kopplar till API-behörigheter och betonar permissions som koncept (Permissions required for fine-grained personal access tokens).

  7. Slå på branch-skydd: PR-krav, granskning och status checks

    • Åtgärd: Skydda publiceringsbranchen (t.ex. main). Miniminivå för ett stabilt CMS-flöde:
      • Kräv PR innan merge
      • Kräv minst 1 godkännande
      • Kräv att status checks passerar (om du har CI/bygge)
    • Verifiering: Försök pusha direkt till main med ett konto som inte ska få göra det. GitHub ska blockera. För PR: skapa en PR utan godkännande och se att merge-knappen blockeras tills krav uppfylls. GitHub beskriver hur skyddade brancher och krav på status checks fungerar (About protected branches and required status checks).

  8. Sätt upp build/deploy så att du kan läsa fel (branch eller Actions)

    • Åtgärd (om “Deploy from a branch”): Säkerställ att den branch/folder du valt faktiskt innehåller byggbara filer. GitHub varnar t.ex. att om du väljer /docs som källa och senare tar bort mappen får du build error för saknad /docs (Configuring a publishing source…).
    • Åtgärd (om “GitHub Actions”): Använd Pages-actions för att paketera och deploya artefakter, t.ex. actions/upload-pages-artifact och actions/deploy-pages (officiella repos) (actions/upload-pages-artifact, actions/deploy-pages).
    • Verifiering: Gå till fliken “Actions” och öppna senaste körningen. Du ska se en lyckad deploy och en publicerad page_url i deploy-steget (Pages-actionen exponerar den som output enligt action-dokumentationen i repot, t.ex. i README för actions/deploy-pages).

  9. Inför en “media-policy”: bilder och länkar som inte bryter builden

    • Åtgärd: Bestäm en enda bildmapp (t.ex. /assets/) och en enkel regel för filnamn (gemener, bindestreck). Om CMS laddar upp bilder: mappa alltid dit. Undvik att sprida bilder i innehållsmappar om du inte har en tydlig orsak.
    • Verifiering: Gör ett testinlägg i CMS med en uppladdad bild. Kontrollera att filen hamnar i rätt mapp och att länken som skrivs i Markdown fungerar på den publicerade sajten.

  10. Gör en kontrollerad “ändringskedja” hela vägen till publicering

  • Åtgärd: Kör en full testcykel: skapa ett nytt inlägg, bifoga en bild, skapa PR (eller commit), få en review, merge:a och verifiera publicerad ändring.
  • Verifiering: Du ska kunna peka på tre bevis i GitHub:
    • PR eller commit som visar diff
    • Godkännande (review) om PR används
    • En lyckad deploy (Pages eller Actions) och att sajten visar ändringen på URL:en

Säkerhet och behörigheter: tokens, branch-skydd och granskningsflöde

Git-baserade CMS blir snabbt en säkerhetsfråga, eftersom verktyget faktiskt skriver till ditt repo. Tre praktiska principer:

  1. Ge verktyget minsta möjliga rättighet Om ditt CMS bara behöver ändra innehållsfiler ska det inte ha admin-rättigheter. GitHub beskriver hur finmaskiga tokens styr åtkomst via permissions (Permissions required for fine-grained personal access tokens). I praktiken: begränsa token till ett repo och bara de scopes/permissions du verkligen behöver för att skapa commits eller PR:ar.

  2. Separera “skriva innehåll” från “publicera till main” Det är här branch-skydd gör jobbet. Med krav på PR och godkänd granskning tvingar du fram ett granskningssteg innan publicering. GitHub förklarar mekaniken: om PR:n saknar godkännande eller om obligatoriska status checks inte är gröna ska merge blockeras (About protected branches and required status checks).

  3. Håll secrets utanför repo Om du bygger i Actions: lägg tokens/nycklar i GitHub Secrets, inte i .yml eller .env som committas. När du väl har secrets är nästa steg att se över vilka workflow-permissions du ger. GitHub rekommenderar att du begränsar GITHUB_TOKEN till minsta nödvändiga rättigheter och du kan ställa detta per jobb i workflow (Assigning permissions to jobs / GITHUB_TOKEN).

Preview och publicering: så verifierar du att ändringar byggs rätt

Det vanligaste felet i GitHub Pages-flöden är inte att CMS:et “inte kan spara”, utan att ändringen aldrig blir synlig för att den inte deployas.

Så här gör du felsökningen konkret:

  • Om du deployar från branch: kontrollera att GitHub Pages faktiskt pekar på rätt branch och rätt folder. GitHub Docs beskriver exakt detta val och nämner att en borttagen /docs-mapp kan ge build error (Configuring a publishing source…).

  • Om du deployar med Actions: öppna senaste workflow-run under “Actions”. Du vill se två tydliga steg i loggarna:

    • uppladdning av artefakt (t.ex. via actions/upload-pages-artifact) (actions/upload-pages-artifact)
    • deploy till Pages (t.ex. via actions/deploy-pages) (actions/deploy-pages)

  • Tolka byggfel i rätt ände: Om du använder Jekyll och får pluginrelaterade fel kan det bero på att GitHub Pages inte stöder en viss plugin. GitHub är tydliga med att Pages inte kan bygga med “unsupported plugins” (Using Jekyll plugins with GitHub Pages). Lösningen är då ofta att bygga själv i Actions (där du kan styra versioner och plugins) och bara deploya statisk output.

När du bör välja bort Git-baserat CMS (och vad du tar i stället)

Git-baserade CMS är bland de bästa innehållshanteringsverktygen för GitHub Pages när du vill ha enkel drift och bra spårbarhet. Men de är inte alltid rätt.

Välj bort Git-baserat CMS om något av detta gäller:

  • Du behöver realtidsredigering i samma dokument av flera personer (Git/PR är sekventiellt; konflikter blir en del av vardagen).
  • Du behöver avancerade redaktörsroller, arbetsflöden och publiceringskalender med många steg och intern governance.
  • Du behöver server-side funktioner som hör ihop med innehåll (personalisering, regler per besökare, komplex sök, integrationslogik) och vill att CMS ska vara navet.

Alternativ som ofta passar bättre då:

  • Ett hostat CMS med inbyggda arbetsflöden och rollhantering (t.ex. klassiska enterprise-CMS eller moderna SaaS-CMS).
  • Ett headless CMS med API-first publicering och en separat front-end deploy (du kan fortfarande bygga statiskt, men du hämtar innehåll via API vid build).

Även om du byter bort Git-baserat CMS kan du behålla några av idéerna i checklistan: tydlig innehållsstruktur, tydliga publiceringsbevis, och ett granskningssteg som går att revidera.
