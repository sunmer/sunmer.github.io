Cloudflare Pages höjer taket till 100 000 filer och byter standard till Node.js v22: när blir det värt att flytta från GitHub Pages?

Om du kör en statisk sajt på GitHub Pages är det lätt att gilla enkelheten: push till ett repo, och sajten är uppdaterad. Men förr eller senare stöter många på samma klassiska väggar: en sajt som växer (fler genererade filer, fler bilder, fler språkversioner) och ett byggsteg som börjar kräva en modernare runtime än vad plattformen ”råkar” ge dig.

Det är precis därför Cloudflare Pages-uppdateringarna är intressanta som migrations-trigger. Cloudflare har både höjt filgränsen till 100 000 filer för betalda planer och gjort Node.js v22 till standard i sin nyare build image. Tillsammans gör det att en typ av GitHub Pages-sajt – den som är ”statisk men stor” och bygger med modern JS-verktygskedja – plötsligt passar bättre på Pages.

Här får du en beslutsgrund: vad som faktiskt ändrats, vad det innebär i praktiken, vilka kompatibilitetsrisker du ska räkna med, och hur du kör en låg-risk pilot där GitHub fortfarande är källan och du inte behöver byta domän direkt.

1) Vad som ändrades i Cloudflare Pages – och vem som märker det

Filgränsen: 20 000 på Free, 100 000 på betalda planer

Cloudflare Pages dokumenterar att Free-planen har en gräns på 20 000 filer per sajt, medan betalda planer kan ha upp till 100 000 filer per sajt. För att aktivera den högre gränsen behöver du dessutom sätta en miljövariabel: PAGES_WRANGLER_MAJOR_VERSION=4 i projektets inställningar i Cloudflare. Det står uttryckligen i Pages ”Limits”-dokumentationen. Cloudflare Pages Limits

Det här är den uppdatering som typiskt påverkar dig om du:

  • genererar mycket innehåll (t.ex. dokumentationssajter med många sidor)
  • har många assets (bilder, kodexempel, språkpacksfiler)
  • bygger med verktyg som spottar ut stora mängder chunkade filer (JS/CSS)

En viktig detalj: Cloudflare kopplar filgränsen till planer och styrning via env-var. Det betyder att en ”flytt till Pages” ibland inte handlar om att byta hosting, utan om att välja en plan som matchar sajtens storlek.

Byggmiljön: Node.js v22 som default i build image v3

Cloudflare har publicerat att Pages-byggar i deras inbyggda build-system nu ger Node.js v22 som standard i den nyare build image (v3), medan Node.js v18 var default tidigare. Cloudflare Changelog: Pages builds now provide Node.js v22 by default

I Pages ”Build image”-dokumentation listas också vilka standardversioner som ingår per build image-version. I v3 är default Node 22.16.0 och du kan override:a via NODE_VERSION, .nvmrc eller .node-version. Cloudflare Pages Build image

Det här märker du om du bygger med t.ex. Next.js, Astro, Vite eller liknande och du vill:

  • slippa pinna Node i CI för att få moderna versioner
  • minska skillnader mellan lokal utveckling (ofta en modern Node LTS) och byggserver

2) Vad 100 000 filer och Node.js v22 betyder i praktiken för byggen

100 000 filer: vad räknas som ”filer” och varför blir du stoppad?

På statiska hostar räknas i praktiken varje uppladdad artefakt: HTML, CSS, JS, bilder, fontfiler, sourcemaps, JSON, med mera. En modern frontend-build kan skapa många fler filer än du tror – särskilt om du har:

  • många routes som förgenereras
  • flera språk (varje språk dubblar ofta antal HTML-filer)
  • stora mängder bilder som optimeras och läggs som separata filer

På Cloudflare Pages är den här gränsen tydlig och dokumenterad som en plattformsgräns. Cloudflare Pages Limits

Praktisk konsekvens: Om du tidigare undvikit Pages på grund av 20 000-filsgränsen (vanligt vid stora docs-sajter) är den invändningen mindre relevant – men bara om du är beredd att köra en betald plan och sätta PAGES_WRANGLER_MAJOR_VERSION=4.

Node.js v22 som default: tre vanliga kompatibilitetsrisker

En ny default-runtime brukar ge två sorters problem: det som går sönder direkt, och det som blir ”subtilt annorlunda”. Cloudflare varnar också för att byggar kan fallera om du tidigare litade på default-versioner och sen migrerar till v3 – då behöver du explicit pinna versioner. Cloudflare Pages Build image

Här är tre typiska risker du bör räkna med när Node v22 blir standard:

  1. Native dependencies och prebuilds Paket som använder native moduler (t.ex. vissa bild-/komprimeringsbibliotek) kan sakna prebuild för exakt Node-version/ABI och då tvingas bygga från källkod. Det kan öka byggtid eller fallera om systemberoenden saknas. Du märker det som node-gyp/kompilatorfel i byggloggar.

  2. Låsfiler och package manager-detektion Cloudflare listar flera begränsningar för v3 build system, bland annat att det inte detekterar Yarn-version från yarn.lock, pnpm-version från pnpm-lock.yaml, och inte heller läser Node/package manager från package.json -> engines. Cloudflare Pages Build image Praktisk konsekvens: du kan behöva sätta versioner explicit (env-var eller filer) även om din repo-konfiguration ”brukar räcka” på andra CI.

  3. Node-kompatibilitet i edge-miljö (om du använder Functions/Workers) För ren statisk hosting spelar det mindre roll. Men om du kör Pages Functions är det i grunden Workers-miljö, och ”Node i build” är inte samma sak som ”Node API i runtime”. Cloudflare beskriver att Functions-requests debiteras som Workers-requests och att statiska assets är fria och obegränsade, vilket är en påminnelse om att du kan hamna i en hybrid där build och runtime har olika constraints. Cloudflare Pages Functions pricing

3) Hur det slår på kostnad och prestanda för stora statiska sajter

Bandbredd och storlek: när GitHub Pages blir trångt

GitHub är tydliga med sina Pages-gränser. En publicerad GitHub Pages-sajt får inte vara större än 1 GB, och det finns en mjuk bandbreddsgräns på 100 GB per månad. Dessutom finns en mjuk gräns på 10 builds per timme (om du använder deras standardflöde; gränsen gäller inte på samma sätt om du bygger och publicerar via ett eget GitHub Actions-workflow). GitHub Docs: GitHub Pages limits

För en portfolio eller en mindre blogg är det sällan ett problem. Men för en större docs-sajt (eller en sajt med mycket media) kan du hamna i ett läge där du måste börja optimera runt gränser, snarare än att bara publicera.

Cloudflare Pages: obegränsade requests till statiska assets, men andra kvoter runtom

Cloudflare skriver att requests till statiska assets är fria och obegränsade (på både free och paid), och att det är Functions som räknas mot Workers-kvoter. Cloudflare Pages Functions pricing

Det här är centralt om din sajt är ”mycket trafik, mest statiskt”. Då kan Cloudflare Pages vara mer förlåtande än en host med tydliga bandbreddstak.

Samtidigt finns andra praktiska gränser du måste väga in:

  • 500 builds per månad på Free och timeout efter 20 minuter. Cloudflare Pages Limits
  • Max 25 MiB per enskild asset (om du har stora PDF:er eller videoklipp behöver du tänka R2 eller annan lagring). Cloudflare Pages Limits

Prestanda handlar inte bara om CDN. För en statisk sajt är det ofta ”time-to-first-byte + caching” som dominerar. Både GitHub Pages och Cloudflare Pages lägger dig bakom en CDN-liknande distribution, men Cloudflare har ett stort edge-nätverk som kan vara en fördel om du har internationell publik. Den faktiska vinsten bör du mäta i pilot (se avsnitt 6).

4) Jämförelse: GitHub Pages vs Cloudflare Pages i ett GitHub-baserat arbetsflöde

Det här är den praktiska kärnan: du vill oftast behålla GitHub som källa (repo, PR-flöde, issues), men du vill att bygg och leverans ska passa din sajt.

När GitHub Pages är bäst (stanna kvar)

GitHub Pages är svårslaget när du vill ha:

  • minimal konfiguration
  • en sajt som är liten (väl under 1 GB) och inte extremt trafikdriven
  • enkel Jekyll/Markdown-publicering utan specialkrav

GitHub publicerar dessutom vilka versionsberoenden de kör för Pages, vilket är relevant om du bygger med Jekyll och behöver matcha deras miljö lokalt. GitHub Pages dependency versions

Det du ofta ”äger själv” på GitHub Pages är att hantera constraints som:

  • storleksgränser (1 GB) och bandbreddssoft-limit (100 GB/månad) GitHub Pages limits
  • byggtimeout på 10 minuter GitHub Pages limits

När Cloudflare Pages blir mer attraktivt (utvärdera flytt)

Cloudflare Pages blir intressant när du har en ”modern JS-byggkedja” och antingen växer i filantal eller vill ha en modernare default-runtime i byggsteget.

Två konkreta triggers som är lätta att koppla till uppdateringarna:

  • du har eller är på väg mot en sajt som kräver fler än 20 000 filer → du kan nå 100 000 om du kör betald plan och sätter PAGES_WRANGLER_MAJOR_VERSION=4 Cloudflare Pages Limits
  • du vill att Cloudflares inbyggda build ska köra Node v22 som standard (och du vill slippa specialscript i CI) Cloudflare Changelog

Dessutom: om du vill lägga ett säkerhetslager nära din statiska sajt kan Cloudflare-ekosystemet (t.ex. WAF på domännivå om du redan använder Cloudflare som DNS/proxy) bli ett argument. Det är inte en ”Pages-funktion” i sig, men det är en del av helheten när du hostar bakom Cloudflare.

5) Beslutsgrund: stanna eller flytta – med risker och friktion

Om du vill fatta ett praktiskt beslut, tänk i fyra kontrollpunkter. Du behöver inte vinna på alla för att en flytt ska vara rimlig.

Kontrollpunkt A: filantal och artefakter

  • Om du tydligt ligger under 20 000 filer och inte växer snabbt: Cloudflare-ändringen hjälper dig inte så mycket.
  • Om du regelbundet bump:ar i filantal (t.ex. docs-sajt med många versioner): Cloudflare Pages kan bli relevant, men förutsätter betald plan och env-var-ändring. Cloudflare Pages Limits

Kontrollpunkt B: byggkedja och Node-version

  • Om du redan bygger i GitHub Actions med pinad Node och bara deployar statiskt till GitHub Pages, har du i praktiken redan ”löst” runtime-frågan.
  • Om du däremot vill använda hostens inbyggda build (för enklare setup), då är en modern default viktig. Cloudflare har explicit gjort v22 till default för Pages builds. Cloudflare Changelog

Kontrollpunkt C: trafikprofil och kostnadsrisk

GitHub Pages har mjuka bandbreddsgränser (100 GB/månad) och storleksgränser (1 GB). GitHub Pages limits

Cloudflare å sin sida är tydliga med att statiska asset-requests är fria och obegränsade (vilket gör modellen attraktiv för statiska sajter med ”spikar”). Cloudflare Pages Functions pricing

Kontrollpunkt D: drift, rollback och ”vem äger vad?”

En flytt innebär nästan alltid ny drift-yta:

  • ny deploy pipeline (även om GitHub är källan)
  • ny felbudget: build-loggar och caching-beteenden kan skilja
  • ny konfig: redirects/headers har egna format och gränser (t.ex. _redirects, _headers). Cloudflare Pages Limits

Det är därför en pilot är så viktig.

6) Så gör du en låg-risk pilot – utan att byta domän direkt

Målet är att testa Cloudflare Pages som ”render target” medan GitHub fortsatt är system of record. Du vill kunna backa ur utan att ändra repo-struktur eller SEO-kritiska URL:er.

Steg 1: skapa ett Pages-projekt kopplat till samma GitHub-repo

Koppla Pages till ditt befintliga repo och använd samma build-kommando som du redan kör lokalt (t.ex. npm ci && npm run build). Poängen här är inte att optimera, utan att få en jämförbar baseline.

Om du vet att du behöver Node-version pinad (för att minska risken för ”default drift”), lägg en .node-version eller .nvmrc i repot, eller sätt NODE_VERSION i Pages settings. Cloudflare beskriver att det går att override:a versioner via env-var och via dessa filer. Cloudflare Pages Build image

Steg 2: om du är beroende av 100 000 filer – slå på det med env-var

Om du misstänker att du ligger över 20 000 filer, gör det här tidigt så att du inte felsöker ”mystiska” deploy-failures.

Sätt PAGES_WRANGLER_MAJOR_VERSION=4 i projektets inställningar. Cloudflare skriver att det behövs för att aktivera den ökade filgränsen på betalda planer. Cloudflare Pages Limits

Steg 3: publicera på en separat preview-domän och jämför

Kör pilot på Cloudflares egna domän (ditt *.pages.dev-namn) och låt din riktiga domän fortsätta peka mot GitHub Pages.

Mät tre saker under en vecka:

  • byggstabilitet: antal misslyckade byggen och typ av fel
  • byggtid: median och p95 (”95:e percentilen”, alltså de långsammaste 5%)
  • prestanda: jämför laddtid och cache headers i webbläsarens DevTools

Du behöver inte en perfekt labbmiljö. Målet är att se om du får färre driftproblem, inte att jaga millisekunder.

Steg 4: lägg in en enkel rollback-plan innan du byter trafik

Rollback ska vara trivialt:

  • Om du byter DNS senare: dokumentera exakt vad som ändras och hur du återställer.
  • Om du använder redirects/headers: håll dem minimala i piloten och skala upp efter att basen fungerar.

Steg 5: byt domän sist (och bara om pilotdata säger att det är värt det)

När piloten är stabil kan du flytta en mindre del av trafiken först (t.ex. en subdomän som staging.example.com) och behålla huvuddomänen på GitHub Pages tills du är trygg.

Det här upplägget gör att du kan prova Pages för det som uppdateringen faktiskt handlar om: större statiska sajter och en modernare byggmiljö, utan att du måste ta alla migrationskostnader på en gång.

Slutsats

Cloudflare Pages-uppdateringarna är inte en generell ”alla borde flytta”-signal. Men de är en tydlig signal för en specifik kategori GitHub Pages-siteägare: du som börjar bli begränsad av storlek/filantal eller som vill ha en mer modern default i byggkedjan utan att lägga mer CI-ansvar på dig själv.

Om du känner igen dig i filantal-problemet är 100 000-gränsen på betalda planer det konkreta skälet att åtminstone köra en pilot. Om du främst är nöjd med GitHub Pages enkelhet och din sajt är liten, är det lika rimligt att stanna – och i stället bygga via GitHub Actions när du behöver mer kontroll.
